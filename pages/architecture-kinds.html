Current architecture

One authority per job

The composition is powerful because the layers remain separate. None needs to become a universal abstraction.

Bundle

The durable, user-owned world: ordinary OKF Markdown documents, links, blobs, conventions, versions, and attribution.

Kind

The controlled domain vocabulary, declared as a Convention document and loaded through one registry.

Relationship

A named semantic assertion from a source Kind to a target Kind, represented by an ordinary link.

Recipe

Today, a data-only, idempotent package of Convention documents adopted explicitly by a bundle.

CLI

The generic action boundary that applies Kind and graph semantics without domain-specific commands.

Page

A bundle-owned projection in a sandbox: content-only or read-only live data, never the source of truth.

Skill / plugin

Procedural guidance and delivery. It may teach use of the model but should not duplicate its semantic authority.

Values

What the architecture is trying to preserve

Open, user-owned state

The knowledge remains inspectable and useful outside one model, vendor, interface, or hosted account.

One semantic authority

CLI, agents, Pages, and future consumers discover the same Convention registry rather than reinterpreting it.

Local-first operation

A bundle can remain local and private indefinitely; remote sharing and storage are additive choices.

Human visibility

Pages project live shared work into task-specific interfaces while preserving the durable record underneath.

Bounded trust

Pages are sandboxed and read-only; mutations use version guards and truthful actor attribution.

Empirical growth

Add operational machinery only when real use demonstrates the missing primitive; avoid declaring prose executable.

Planned

Self-description makes the model transferable

The existing registry transfers structure. Optional semantic descriptions would let an unfamiliar agent understand intent at the operational moment where it chooses a Kind, field, relationship, or enum value.

Kind description
What an instance represents, when to create it, and how it differs from neighboring Kinds.
Field descriptions
What information belongs in each field. A valid value should not be semantically misapplied.
Relationship descriptions
What an outbound link asserts and in which direction—ownership, evidence, decomposition, dependency, or something else.
Enum descriptions
The domain meaning of lifecycle, confidence, approval, quality, and publication values—not merely their spelling.
Sections + examples
Deferred until dogfooding shows that the first four layers do not prevent real misuse. Examples carry maintenance and sedimentation risk.

Better agent authoring

Meaning appears during Kind discovery and creation, reducing structurally valid but conceptually wrong records.

More faithful generated Pages

A Page-generating agent can choose labels, grouping, and graph semantics from one model rather than infer from names.

Portable operating models

A recipient can inherit domain meaning with a recipe instead of requiring the author to explain the vocabulary live.

Human confirmation

A proposed domain model becomes reviewable because people can confirm definitions, not just names and field shapes.

Coherent skills

Skills teach procedures and judgment while querying the Kind authority for compact semantic definitions.

Safer future automation

Descriptions can inform agent judgment, but remain non-executable guidance; deterministic rules still require code.

Hypothesis

A portable cognitive ecosystem

A future recipe could package a domain's concepts and relationships, working practices, optional Pages, examples, focused skills, and deterministic checks. Humans and agents would enter an evolvable operating environment rather than a fixed application.

Not yet proven

Founder-to-founder transfer must show that one person can package a real operating model and another can install and use it without live explanation.

Not a general workflow engine

Operational moments may invoke narrow deterministic primitives, but arbitrary transitions, triggers, and business rules are deferred.

Not ontology maximalism

The design intentionally excludes scalar coercion, cardinality, computed fields, arbitrary validation expressions, and embedded prompts.

Honest boundaries

What this does not claim

Implemented now

  • Plain bundle-owned Convention docs and one Kind registry
  • Generic Kind-aware CLI creation, validation, queries, and typed links
  • Version tokens, compare-and-swap writes, attribution, and history
  • Sandboxed Pages with explicit content-only or read-only bridge capability
  • Live invalidation and Page-to-Page registered navigation
  • Data-only, expect-absent recipe application

Not implemented now

  • Machine-readable Kind, field, relationship, or enum descriptions
  • Page mutation controls or human identity authorization
  • A durable, ordered, replayable event journal
  • Complete recipe packages with automatic skill/Page/hook adoption
  • Recipe upgrade, migration, dependency, or rollback machinery
  • A general workflow or ontology engine
